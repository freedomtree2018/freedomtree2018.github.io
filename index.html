AI工具-GitHub Copilot的使用教程

🚀 第一步：安装与激活

安装插件

在 VS Code 的扩展市场搜索 “GitHub Copilot” 并安装（选择官方插件）。

注意：

  • 如果搜索后第一个插件显示启用/禁用AI功能而非安装按钮，说明插件已安装，只需启用即可。
  • 确保安装的是官方 GitHub Copilot 插件，避免第三方仿冒。

检查安装状态

  1. 右下角状态栏： 查看是否有 Copilot 图标（类似两个气泡或 GitHub 猫头）。如果有，插件已安装。
  2. 侧边栏： 点击左侧活动栏的 Copilot 图标（对话气泡形状）。如果能打开聊天窗口，说明正常。

提示： 如果侧边栏无图标，尝试重启 VS Code 或检查扩展是否启用。

激活与登录

  • 如果右下角图标无禁止符号，已激活。
  • 如果有禁止符号，点击图标，选择“Sign in to GitHub”登录。
  • 若登录失败，访问 GitHub Copilot 官网，登录账号并激活免费计划（Copilot Free）。

常见问题： 如果点击图标出现报错，刷新 VS Code 或重新登录 GitHub。

测试激活

打开代码文件（如 .js），写注释如 // 创建一个红色的按钮，按回车查看灰色建议。若出现，按 Tab 采纳，说明激活成功。

⚙️ 开始使用

安装并激活后：

  1. 登录 GitHub 账号： 右下角提示或侧边栏登录。
  2. 确认订阅： GitHub Copilot 提供免费层级（Copilot Free），包括每月 2000 次代码补全。学生和开源维护者可免费升级。普通用户可试用后订阅（个人版约 $10/月）。
  3. 开始编码：
    • 代码补全： 写代码时，Copilot 以灰色字体建议，按 Tab 采纳。
    • AI 聊天： 点击侧边栏 Copilot 图标或按 Ctrl+L，提问或生成代码。

前端AI工具学习心得

前端AI工具使用心得：提升开发效率的利器

引言

在人工智能技术飞速发展的今天，前端开发领域也迎来了诸多AI工具的革新。作为一名前端开发者，我在过去半年中深入体验了多款AI辅助工具，这些工具不仅显著提升了我的开发效率，更改变了传统的编码思维模式。本文将分享我在使用这些工具过程中的真实体验和心得体会。

主流前端AI工具概览

1. GitHub Copilot

使用体验：
GitHub Copilot是我最早接触的AI编程助手，集成在VS Code中后，它能够根据当前代码上下文智能推荐下一行代码。在编写React组件时，Copilot经常能准确预测我想要的JSX结构，大大减少了样板代码的编写时间。

亮点功能：

  • 实时代码补全，支持多种编程语言
  • 根据注释生成代码逻辑
  • 快速生成测试用例

使用建议：
建议在编写重复性较高的代码时使用，如表单验证、API调用封装等，但对于核心业务逻辑仍需仔细审查生成的代码。

2. Tabnine

使用体验：
Tabnine的本地化部署选项让我印象深刻，它在保护代码隐私方面做得很好。在处理敏感项目时，我更倾向于使用Tabnine，因为它可以在本地运行，避免代码上传到云端。

亮点功能：

  • 支持离线模式
  • 深度学习模型针对代码上下文理解
  • 轻量级，对系统资源占用小

3. Codeium

使用体验：
作为免费的AI编程助手，Codeium的表现超出预期。在团队协作项目中，它帮助新成员快速上手项目结构，通过自然语言查询代码库，大大降低了学习成本。

实际应用场景体验

场景一：组件开发加速

在使用React开发时，我经常需要创建功能相似的组件。AI工具能够根据我已有的组件模式，快速生成新的组件框架。例如，当我需要创建一个数据表格组件时，只需输入注释”// 创建用户数据表格组件”，AI就能自动生成包含分页、排序、筛选等基础功能的组件骨架。

场景二：调试辅助

遇到难以定位的bug时，我尝试将错误信息和相关代码片段输入给AI工具，它能够帮助分析可能的错误原因，并提供修复建议。虽然不能完全替代人工调试，但确实能提供有价值的思路。

场景三：技术文档生成

AI工具在生成API文档、组件说明等方面表现出色。通过分析代码结构，它能自动生成清晰的技术文档，这在团队协作和项目交接时非常有用。

使用心得与建议

正面体验

  1. 效率显著提升：重复性代码编写时间减少50%以上
  2. 学习成本降低：新框架、新技术的学习曲线变得平缓
  3. 代码质量改善：AI工具能发现一些潜在的代码质量问题

需要注意的问题

  1. 代码审查不可少：AI生成的代码可能存在安全漏洞或逻辑错误
  2. 过度依赖风险：长期使用可能影响独立编程能力
  3. 版权问题：需注意生成代码的版权归属

最佳实践建议

  1. 保持批判性思维：将AI工具视为助手而非替代品
  2. 持续学习：在使用工具的同时，仍要深入理解技术原理
  3. 团队协作：在团队中统一AI工具的使用规范

未来展望

随着AI技术的不断进步，前端开发必将迎来更多创新工具。我认为未来AI工具将更多地融入开发流程的各个环节，从需求分析、架构设计到代码实现和测试维护，形成完整的智能开发体系。

结语

AI工具为前端开发带来了革命性的变化，但工具始终是工具，开发者的核心价值在于对业务的理解、架构的设计和问题的解决能力。合理利用AI工具，保持持续学习的态度，我们才能在这个快速变化的时代中保持竞争力。

使用AI工具的开发者，不是被AI替代的人，而是懂得如何与AI协作的人。

本文为个人使用心得分享，工具效果可能因具体场景而异。

let和const

let

let: 声明变量。
特点：

  • 不存在变量提升
  • 不允许重复声明
  • 存在暂时性死区，不能在声明前就使用变量

const

const: 声明一个只读的常量。
特点：

  • 一旦声明，这个值不能被改变
  • 声明时必须赋值

扩展：let和var和const的区别

特性 var let const
作用域 函数作用域 块级作用域 块级作用域
变量提升 提升并初始化为 undefined 提升但不初始化 (存在暂时性死区) 提升但不初始化 (存在暂时性死区)
重复声明 允许 不允许 不允许
重新赋值 允许 允许 不允许 (绑定不可变)
全局对象属性 会挂载到 window 不会 不会

变量提升: var 声明的变量会被提升到作用域顶部，并初始化为 undefined。因此，在声明前访问它不会报错，但会得到 undefined
暂时性死区: let 和 const 声明的变量虽然也会被提升，但在声明语句执行前，它们处于“暂时性死区”（Temporal Dead Zone, TDZ）。在此期间访问它们会抛出 ReferenceError 错误，这强制了“先声明，后使用”的良好习惯。

箭头函数
箭头函数更适用于那些本来需要匿名函数的地方
箭头函数不能用作构造函数
没有自己的this、arguments、super和new.target

vue路由模式

Posted on 2021-03-31 Edited on 2026-04-24

vue-router路由对象中，路由有两种模式：hash和history，而默认的是hash模式

路由的核心：改变视图的同时不会像后端发出请求

hash路由

原理：hash原理其实是浏览器监听了一个onhashchange的方法来改变页面的跳转的
特点：hash虽然出现在url中，但不会包含在HTTP请求中，对后端完全没有影响，因此改变hash不会重新加载页面。
缺点：只能改变#后面的参数来实现路由的跳转。

因为hash发生变化的url都会被浏览器记录下来，从而会发现浏览器的前进后退都可以用。

history路由

原理：利用了HTML5 的History方法。 中新增的，不支持老旧浏览器。

特点1：history模式包括back、forward、go三个方法，对应浏览器的前进，后退，跳转操作。

特点2：history模式可以修改替换历史状态，包括了pushState,replaceState两个方法，这两个方法接受三个参数：stateObj,title,url

通过pushstate把页面的状态保存在state对象中，当页面的url再变回这个url时，可以通过event.state取到这个state对象，从而可以对页面状态进行还原，这里的页面状态就是页面字体颜色，实际滚动条位置，阅读进度，组件的开关的这些页面状态都可以存储到state的里面

缺点：history模式在dev阶段正常，但是在部署服务器的时候，需要在ng上配置相应的正向代理跳转，否则跳转会出错。
